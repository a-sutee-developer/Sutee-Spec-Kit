根据需求文档（.idea/suteeCopilot/spec/requirement.md）和设计文档（.idea/suteeCopilot/spec/design.md）创建一个可执行的实施计划，其中包含编码任务的检查清单。任务文档应该基于需求文档和设计文档，因此请确保它首先存在。
Constraints:
如果 '.idea/suteeCopilot/spec/tasks.md' 文件不存在，模型必须创建它
如果用户指出需要对设计进行任何更改，模型必须修改design.md，并提示用户检查修改后的文档，在用户确认无误后再修改task.md
如果用户指出我们需要额外的需求，模型必须修改requirement.md，并提示用户检查修改后的文档，在用户确认无误后再修改task.md
模型必须在 '.idea/suteeCopilot/spec/tasks.md' 创建实施计划
模型必须在创建实施计划时使用以下具体说明：
将功能设计转换为一系列提示（prompts），供代码生成 LLM 以测试驱动的方式实施每个步骤。优先考虑最佳实践、增量进展和早期测试，确保在任何阶段都没有复杂性的大跳跃。确保每个 prompt 都建立在之前的 prompts 之上，并以将事物连接在一起结束。不应该有任何悬空或孤立的代码未集成到之前的步骤中。仅关注涉及编写、修改或测试代码的任务。

模型必须将实施计划格式化为编号的复选框列表，最多两级层次结构：

顶级项目（如 epics）应仅在需要时使用
子任务应使用十进制表示法编号（例如 1.1、1.2、2.1）
每个项目必须是一个复选框
首选简单结构

模型必须确保每个任务项目包含：

作为任务描述的明确目标，涉及编写、修改或测试代码
作为任务下子项的附加信息
对需求文档中需求的具体引用（引用细粒度的子需求，而不仅仅是用户故事）

模型必须确保实施计划是一系列离散的、可管理的编码步骤

模型必须确保每个任务引用需求文档中的具体需求

模型不得包含设计文档中已涵盖的过多实施细节

模型必须假设所有上下文文档（功能需求、设计）在实施期间都可用

模型必须确保每个步骤都在之前的步骤上增量构建

模型应该在适当的地方优先考虑测试驱动开发

模型必须确保计划涵盖可以通过代码实施的设计的所有方面

模型应该排序步骤以通过代码早期验证核心功能

模型必须确保实施任务涵盖所有需求

模型必须仅包含可以由编码 agent 执行的任务（编写代码、创建测试等）

模型不得包含与用户测试、部署、性能指标收集或其他非编码活动相关的任务

模型必须专注于可以在开发环境中执行的代码实施任务

模型必须通过遵循以下准则确保每个任务可由编码 agent 执行：

任务应涉及编写、修改或测试特定的代码组件
任务应指定需要创建或修改的文件或组件
任务应足够具体，以便编码 agent 可以在没有额外澄清的情况下执行它们
任务应关注实施细节而不是高级概念
任务应限定在特定的编码活动范围内（例如，"Implement X function" 而不是 "Support X feature"）
模型必须明确避免在实施计划中包含以下类型的非编码任务：

用户验收测试或用户反馈收集
部署到生产或暂存环境
性能指标收集或分析
运行应用程序以测试端到端流程。但是，我们可以编写自动化测试以从用户角度测试端到端
用户培训或文档创建
业务流程变更或组织变更
营销或沟通活动
